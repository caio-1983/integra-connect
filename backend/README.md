# Integra Connect — AI Runtime

Sprint 010 backend: orchestrates agents, models, providers, tools, knowledge and
telemetry for Integra Connect. Runs independently of the Vite frontend, on its
own VPS. The frontend only ever calls `POST /v1/agents/:agentId/chat` — it
never knows about OpenAI, models, or tools (see the root plan doc for the
full architecture rationale).

## Local development

```bash
cd backend
npm install
cp .env.example .env   # fill in OPENAI_API_KEY and a GATEWAY_API_KEY
npm run dev            # Fastify on http://localhost:8787, Swagger UI at /docs
```

Smoke test (works even with a placeholder `OPENAI_API_KEY` — you should get a
clean error, not a crash):

```bash
curl -X POST http://localhost:8787/v1/agents/atendimento/chat \
  -H "Authorization: Bearer <your GATEWAY_API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"conversationId":"conv1","contactId":"ct1","channel":"whatsapp","messages":[{"fromType":"user","content":"oi"}]}'
```

## Deploying to your VPS (manual — no CI/CD wired up yet)

### Option A — Docker

```bash
scp -r backend/ user@your-vps:/opt/ai-runtime
ssh user@your-vps
cd /opt/ai-runtime
cp .env.example .env && nano .env   # fill in real secrets
docker build -t ai-runtime .
docker run -d --name ai-runtime --restart unless-stopped \
  --env-file .env -p 8787:8787 ai-runtime
```

### Option B — PM2 (no Docker)

```bash
scp -r backend/ user@your-vps:/opt/ai-runtime
ssh user@your-vps
cd /opt/ai-runtime
cp .env.example .env && nano .env
npm ci
npm run build
npm i -g pm2   # if not already installed
pm2 start dist/index.js --name ai-runtime
pm2 save
```

### nginx reverse proxy + TLS (sample)

```nginx
server {
    listen 80;
    server_name ai.yourdomain.com;

    location / {
        proxy_pass http://127.0.0.1:8787;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Then `sudo certbot --nginx -d ai.yourdomain.com` for TLS.

Finally, point the frontend's `.env` at the deployed URL:

```
VITE_AI_GATEWAY_URL=https://ai.yourdomain.com
VITE_AI_GATEWAY_KEY=<same GATEWAY_API_KEY as backend/.env>
```

## What's implemented (Sprint 010)

Real OpenAI provider, per-agent model resolution, real tool-calling (LLM ⇄
tool round trip), a pipeline-based `AgentRuntime` (Context → Memory →
Knowledge → Prompt → Model+Tools → Post-processing → Handoff → Response),
pass-through AI middleware slots (Guardrails/Moderation/ContextLimit/
ToolPermission — ready to enforce policy later without touching the
runtime), and in-memory/log-only telemetry via an event bus.

## WhatsApp via Evolution API (Sprint 011)

Real WhatsApp, fully decoupled from the AI Runtime and multi-channel-ready:

```
Evolution → POST /webhooks/:provider/:secret (thin) → EventBus
  → ChannelOrchestrator (normalize, version-adaptive) → InboundMessageReceived
  → ConversationService → ConversationRepository (Supabase) → mode gate → AgentRuntime
  → OutboundMessageRequested → OutboundMessageService → Evolution → WhatsApp
```

- **Version auto-detect:** on first call the backend hits Evolution's `GET /`,
  reads `version`, and adapts create/webhook/send payloads to v1 or v2 (see
  `channels/evolution/V1Adapter.ts` / `V2Adapter.ts`). Nothing is assumed.
- **Persistence:** `ConversationRepository` is the ONLY module that touches
  Supabase (service-role, `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`),
  writing the same `contacts`/`conversations`/`messages` shapes as the legacy
  `whatsapp-webhook`, so the existing frontend realtime shows messages live
  with no frontend changes. It deliberately does **not** enqueue
  `nina_processing_queue` (that would fire the old Meta orchestrator).
- **AI reply** is generated by this backend's own `AgentRuntime` and sent back
  through Evolution — auto-reply only when the conversation is in autonomous
  mode (DB `status='nina'`); a human/paused conversation is persisted but the
  AI stays silent.

### Additional required env (see `.env.example`)
`EVOLUTION_API_URL`, `EVOLUTION_API_KEY` (global apikey), `PUBLIC_BASE_URL`
(this backend's public HTTPS URL, so Evolution can reach the webhook),
`EVOLUTION_WEBHOOK_SECRET`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`.

### Connecting a number
Frontend → Gestão de Canais → WhatsApp → **Conectar via Evolution** creates the
instance, auto-registers the webhook at
`${PUBLIC_BASE_URL}/webhooks/evolution/${EVOLUTION_WEBHOOK_SECRET}`, shows the
QR, and polls until connected. `PUBLIC_BASE_URL` **must be publicly reachable
by your Evolution server** (behind the nginx/TLS setup above) or webhooks will
never arrive.
