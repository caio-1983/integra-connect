# Integra Connect

# UX - Experiência do Usuário

Versão: 1.0

---

# Objetivo

A experiência do Integra Connect deve permitir que qualquer usuário consiga utilizar a plataforma sem treinamento técnico.

O sistema deverá ser intuitivo.

Simples.

Objetivo.

Consistente.

Toda decisão de UX deverá reduzir esforço cognitivo.

Nunca aumentar.

---

# Filosofia

O usuário não compra software.

O usuário compra resultado.

Ele deseja resolver um problema.

Não aprender um sistema.

Quanto menos o usuário precisar pensar para executar uma tarefa, melhor será a experiência.

---

# Princípio Nº 1

## Simplicidade acima de funcionalidade

Sempre que existir conflito entre:

Mais funcionalidades

e

Mais simplicidade

A simplicidade deverá vencer.

---

# Princípio Nº 2

## A operação vem antes da configuração

O usuário abre o sistema para trabalhar.

Não para configurar.

A configuração deverá ocupar o menor espaço possível.

Sempre que possível deverá ficar dentro do módulo Configurações.

Nunca dominar a tela inicial.

---

# Princípio Nº 3

## O usuário deve entender a tela em menos de 30 segundos

Se uma pessoa abrir uma tela pela primeira vez ela deverá responder rapidamente:

Onde estou?

O que essa tela faz?

O que devo fazer agora?

Caso isso não aconteça a experiência precisa ser revista.

---

# Princípio Nº 4

## Um clique = uma decisão

Cada ação importante deverá possuir um único botão principal.

Evitar excesso de opções.

Evitar múltiplos botões concorrendo pela atenção.

---

# Princípio Nº 5

## Mostrar apenas o necessário

Não apresentar todas as funcionalidades ao mesmo tempo.

Utilizar revelação progressiva.

O usuário descobre recursos conforme avança.

Nunca através de telas complexas.

---

# Princípio Nº 6

## A Inteligência Artificial trabalha nos bastidores

A IA não deve dominar a interface.

Ela deve aparecer apenas quando gerar valor.

Exemplos:

Sugestão de resposta.

Resumo.

Automação.

Classificação.

Nunca como protagonista da experiência.

---

# Princípio Nº 7

## Consistência absoluta

Todo comportamento deve ser previsível.

Se um botão funciona de determinada maneira em uma tela.

Ele deverá funcionar igual em todas as outras.

O usuário nunca deverá reaprender a interface.

---

# Princípio Nº 8

## A interface deve transmitir controle

Ao abrir qualquer módulo o usuário deve sentir que possui domínio da operação.

Nunca sensação de caos.

Nunca sensação de sistema complicado.

---

# Hierarquia Visual

Toda tela deverá seguir esta ordem de leitura.

1. Contexto

Onde estou?

---

2. Indicadores

Como está minha operação?

---

3. Conteúdo principal

O que preciso fazer?

---

4. Ações

Quais decisões posso tomar?

---

5. Configurações

Somente quando necessário.

---

# Carga Cognitiva

Evitar:

Muitos cards.

Muitos botões.

Muitas cores.

Muitos gráficos.

Muitas informações simultâneas.

Preferir poucas informações extremamente úteis.

---

# Leitura

Toda tela deverá poder ser compreendida rapidamente.

Evitar blocos grandes.

Utilizar títulos claros.

Subtítulos objetivos.

Agrupamentos naturais.

Espaçamento adequado.

---

# Objetivo Final

O usuário nunca deverá procurar informações.

As informações deverão naturalmente chamar sua atenção.


---

# Dashboard

O Dashboard é o Centro de Operações do Integra Connect.

Sua função não é mostrar gráficos.

Sua função é responder rapidamente:

• Como está minha operação?

• Existem problemas?

• O que precisa da minha atenção?

• O que devo fazer agora?

O Dashboard nunca deverá parecer um painel administrativo.

---

# Prioridade das Informações

Toda tela deverá seguir esta ordem.

1. Contexto

2. Estado da operação

3. Conteúdo principal

4. Ações

5. Configurações

Configurações nunca deverão ocupar posição de destaque.

---

# KPIs

KPIs devem responder perguntas.

Nunca apenas mostrar números.

Exemplos:

Conversas aguardando

Tempo médio

SLA

Canais ativos

Agentes online

Automações executadas

Nunca utilizar KPIs genéricos apenas para preencher espaço.

---

# Cards

Cada card deverá responder apenas uma pergunta.

Nunca misturar assuntos.

Evitar:

Card enorme contendo muitas informações.

Preferir:

Diversos cards pequenos.

Cada um com um único objetivo.

---

# Tabelas

As tabelas deverão ser extremamente limpas.

Sempre priorizar:

Pesquisa

Filtros

Ordenação

Paginação

Ações rápidas

Nunca utilizar excesso de colunas.

Sempre mostrar apenas as informações necessárias.

---

# Pesquisa

Sempre que existir grande volume de dados.

A pesquisa deverá aparecer antes dos filtros.

A pesquisa deverá aceitar linguagem simples.

Exemplo:

Nome

Telefone

Empresa

E-mail

Etiqueta

Nunca exigir formatos específicos.

---

# Filtros

Filtros deverão ser simples.

Nunca esconder informações importantes.

Filtros avançados deverão permanecer recolhidos.

---

# Formulários

Formulários deverão possuir:

Poucos campos.

Agrupamentos lógicos.

Boa separação.

Rótulos claros.

Mensagens objetivas.

Nunca apresentar dezenas de campos simultaneamente.

---

# Botões

Cada tela deverá possuir apenas uma ação principal.

Ela deverá possuir maior destaque visual.

As demais ações deverão possuir menor prioridade.

Nunca criar competição entre botões.

---

# Modais

Utilizar modais apenas para tarefas rápidas.

Nunca colocar grandes formulários dentro de modais.

Quando necessário utilizar páginas completas.

---

# Feedback

Toda ação deverá possuir retorno imediato.

Exemplos:

Salvando...

Salvo com sucesso.

Erro ao salvar.

Conectado.

Desconectado.

Nunca deixar o usuário em dúvida.

---

# Estados Vazios

Toda tela vazia deverá orientar o usuário.

Nunca apenas informar que não existem registros.

Exemplo:

Ainda não existem canais conectados.

Conecte seu primeiro canal para começar a utilizar a plataforma.

---

# Erros

Mensagens de erro deverão explicar:

O que aconteceu.

Por que aconteceu.

Como resolver.

Nunca utilizar mensagens técnicas.

Nunca mostrar stack trace.

Nunca mostrar códigos internos.

---

# Confirmações

Somente solicitar confirmação quando existir risco real.

Exemplo:

Excluir.

Cancelar.

Remover.

Desconectar.

Evitar confirmar ações simples.

---

# Onboarding

O onboarding deve ensinar.

Nunca interromper a operação.

Após a implantação inicial.

O onboarding deverá desaparecer naturalmente.

Nunca ocupar espaço permanente na Home.

---

# Navegação

A navegação deverá ser previsível.

O usuário nunca deverá procurar funcionalidades.

Cada módulo deverá possuir responsabilidades claras.

---

# Consistência

Componentes iguais.

Comportamentos iguais.

Mensagens iguais.

Espaçamentos iguais.

Ícones iguais.

Terminologia igual.

Nunca criar exceções.

---

# Linguagem

Sempre utilizar linguagem empresarial.

Preferir:

Canal

Equipe

Atendimento

Contato

Conhecimento

Relatórios

Automações

Evitar:

Prompt

LLM

Token

Webhook

Workflow

Trigger

Instance

API Key

---

# Performance Percebida

A interface deverá transmitir velocidade.

Mesmo quando existir processamento interno.

Sempre fornecer feedback visual.

Evitar telas congeladas.

Utilizar skeletons.

Indicadores de carregamento.

Transições suaves.

---

# Regra de Ouro

Toda nova funcionalidade deverá responder positivamente às seguintes perguntas:

Ela reduz esforço?

Ela reduz cliques?

Ela melhora organização?

Ela melhora produtividade?

Ela é intuitiva?

Ela fortalece a identidade do produto?

Caso alguma resposta seja negativa.

A funcionalidade deverá ser reavaliada.

---

# Experiência Final

O sucesso do Integra Connect será medido pela facilidade de uso.

O usuário nunca deverá dizer:

"Preciso aprender esse sistema."

Ele deverá dizer:

"Esse sistema parece ter sido feito para minha empresa."
